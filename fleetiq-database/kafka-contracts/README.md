# FleetIQ Kafka Event Contracts & Topic Guidelines

This directory houses the Avro schemas (`.avsc`) defining the core asynchronous message streams for the FleetIQ Fleet Management Platform.

---

## 1. Topic Naming Convention

All topics must strictly adhere to the following template:

```
{environment}.fleetiq.{domain}.{event_type}.v{version}
```

- **`environment`**: `dev`, `staging`, `prod`
- **`domain`**: Functional area, e.g. `telemetry`, `fuel`, `health`, `alert`, `trip`, `driver`, `ev`, `notification`
- **`event_type`**: Noun-verb event descriptor, using kebab-case (e.g. `gps-reading`, `theft-detected`, `dtc-triggered`)
- **`version`**: Major version number prefix (e.g. `v1`, `v2`)

### Examples
- `prod.fleetiq.telemetry.gps-reading.v1`
- `prod.fleetiq.fuel.theft-detected.v1`
- `prod.fleetiq.health.dtc-triggered.v1`
- `prod.fleetiq.alert.dispatched.v1`

## 3. Schema Evolution & Compatibility Rules

We enforce Confluent Schema Registry **`BACKWARD`** compatibility mode.

### Rules for Schema Modifiers:
1. **Adding Fields**: Allowed only if a **`default`** value is defined. This allows older consumers (who do not know about the new field) to parse messages generated by newer producers.
2. **Removing Fields**: Not allowed. To deprecate a field, keep the field as optional (union with null) and default it to null in the producer.
3. **Modifying Types**: Changing field types is prohibited. Instead, create a new field with a new name and default value.
4. **Breaking Changes**: Any breaking change (e.g. changing mandatory field types or deleting fields) requires a new schema filename and a **major version increment** in the topic name (e.g. `v1` -> `v2`).

## 4. Consumer Group Naming Convention

All consumers must be named consistently to simplify auditing and metrics:

```
{service_name}.{domain}.{purpose}
```

- **`service_name`**: Name of the microservice, e.g. `telemetry-ingester`, `alert-engine`
- **`domain`**: Domain being consumed, e.g. `telemetry`, `fuel`
- **`purpose`**: Functional action, e.g. `save-db`, `evaluate-geofences`

### Examples
- `telemetry-ingester.telemetry.save-timescaledb`
- `alert-engine.telemetry.evaluate-speeding`
- `notification-dispatcher.alert.send-sms`

## 5. Dead Letter Queue (DLQ) & Error Handling

To ensure high-throughput processing without data loss:

- **DLQ Topic Name**: `{original_topic_name}.dlq`
  - Example: `prod.fleetiq.telemetry.gps-reading.v1.dlq`
- **Routing**: If a consumer fails to parse or process a message after 3 retries (using exponential backoff), the message is wrapped with an error header containing:
  - `failed_at`: ISO8601 timestamp
  - `exception_message`: Reason for failure
  - `consumer_group`: Group name that failed
  and published to the corresponding DLQ.

---

## 6. Exactly-Once Semantics (EOS)

To prevent duplicate distance, fuel, or billing logs:

- **Producers**: Must enable idempotency and transactional guarantees:
  ```properties
  enable.idempotence=true
  acks=all
  max.in.flight.requests.per.connection=1
  ```
- **Consumers**: Must set isolation levels to read only committed transactions:
  ```properties
  isolation.level=read_committed
  ```
